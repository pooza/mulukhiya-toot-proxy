# リリースノート テンプレート

`gh release create vX.Y.Z --target main --title "X.Y.Z" --notes-file <(...)` で使う定型フォーマット。5.18.0 / 5.19.0 形式を基準にしている。

## いつ使うか

- マイナー / パッチリリース（ホットフィックス含む）の `gh release create` 時
- 過去のリリースノート（5.11.0 以前）も参考になる

## 必須セクション

1. **概要** — 1〜2 文。マイルストーンへのリンクを必ず添える
2. **変更内容** — `### 機能追加` / `### セキュリティ` / `### バグ修正` / `### リファクタ` / `### 廃止` / `### メンテナンス` の小見出しに分け、Issue 番号付きで列挙。該当しない見出しは省略可
3. **アップグレード手順** — Wiki 更新手順 + 4.x 系アップグレードガイドへのリンク。起動スクリプト変更の有無も明記

## 任意セクション

- **運用向け設定変更** — `config/local.yaml` の追加設定が必要な場合
- **既知の問題** — リリース時点で残っているバグ、次リリースで対応予定の項目
- **謝辞** — 外部 contributor がいる場合

## テンプレート本体

```markdown
## 概要

<1〜2 文の要約>。
[X.Y.Z マイルストーン](https://github.com/pooza/mulukhiya-toot-proxy/milestone/<番号>?closed=1)

## 変更内容

### 機能追加

- **#XXXX <Issue タイトル>** — <要約。capsicum 連携や運用観察の背景があれば添える>

### セキュリティ

- **#XXXX <Issue タイトル>** — <CVE / GHSA 等あればリンク>

### バグ修正

- **#XXXX <Issue タイトル>** — <根本原因と修正方針>

### リファクタ

- **#XXXX <Issue タイトル>** — <親 Issue があれば添える>

### 廃止

- **#XXXX <機能名>の全廃止** — <廃止理由>

### メンテナンス

- **bundle update** — <主要な更新があれば併記>

## 運用向け設定変更

<該当時のみ。`config/local.yaml` の追記例を YAML ブロックで示す>

## 既知の問題

<該当時のみ。次リリースで対応予定の項目を箇条書き>

## アップグレード手順

[更新手順](https://github.com/pooza/mulukhiya-toot-proxy/wiki/%E6%9B%B4%E6%96%B0%E6%89%8B%E9%A0%86)を参照してください。
4.x系からのアップグレードは[4.x → 5.0 アップグレードガイド](https://github.com/pooza/mulukhiya-toot-proxy/wiki/4.x-%E2%86%92-5.0-%E3%82%A2%E3%83%83%E3%83%97%E3%82%B0%E3%83%AC%E3%83%BC%E3%83%89%E3%82%AC%E3%82%A4%E3%83%89)も参照してください。

<起動スクリプト変更がない場合>
本リリースに起動スクリプトの変更はありません。通常の更新手順（git pull + bundle install + サービス再起動）で適用できます。

<起動スクリプト変更がある場合>
本リリースは rc.d スクリプト / systemd unit に変更があります。デプロイ時に [config/sample/](../config/sample/) を参照して更新してください。
```

## 注意

- セキュリティ更新（gem パッチ）は実害がなくてもリリースノートに記載する
- 「アップグレード手順」のリンク 2 種は毎回必ず含める（4.x 系利用者向けは当面継続）
- ホットフィックス（5.x.1 等）でも同フォーマットを使う。「概要」は「ホットフィックス。〜の回帰を修正」程度で短く
- 設定キーは Ginseng のスラッシュ表記（`/ruby/jit` 等）で統一
